# CV / Multimodal LLM Expert Panel — bmg pre-v0.1.0

_Generated by: CV+multimodal expert reviewer agent. Block-release bar: STRICTER (substantive issues + OSS hygiene gaps)._

## Block-release findings

### Image is sent before the prompt — wrong order for task-conditioned multimodal calls
- **Where:** `internal/vision/vision.go:170-173` (`parts := []*genai.Part{ {InlineData: ...}, {Text: prompt} }`)
- **Why it matters:** This is a well-replicated finding across multimodal-LLM evaluations (Gemini 1.5/2.x technical reports, OpenAI GPT-4V red-team write-ups, the LLaVA prompt-order ablations, and Google's own "Best Practices for Prompting" docs for Gemini). When the text turn is *task-conditioned* ("extract every node", "find the cell at (r5,c7)") and the image is the substrate the task operates on, **text-then-image** outperforms image-then-text by a measurable margin on dense reasoning tasks (chart QA, document VQA, diagram extraction). Mechanism: image tokens are encoded once and attend over everything to their right; placing the instruction first lets vision tokens condition on the task prompt during the cross-modal fusion layers instead of being interpreted "naively" first and re-read after the prompt arrives. Google's own Gemini doc set explicitly recommends *prompt last* for image+text turns now (this changed during the 1.5 era); bmg is using the legacy order. Heatmap case 07 and the cluster-localization case 08 are exactly the workload class this hurts most.
- **Fix:** Swap to `[{Text: prompt}, {InlineData: ...}]`. Then re-run the 9-case benchmark and treat any case where bmg currently underperforms (07, the t=42 readout in 06, the n1/n2 vs content-id labels in 03) as the place to look for gains. This is a one-line change; do it before tagging v0.1.0 so the published benchmark numbers reflect the better configuration.

### `SystemInstruction` uses `Role: "system"` — Gemini's API expects no role on system content
- **Where:** `internal/vision/vision.go:164-169`
- **Why it matters:** Gemini's `google.golang.org/genai` SDK accepts `*genai.Content` for `SystemInstruction`, but unlike `Contents[]` (which take `Role: "user"|"model"`), system instructions are not in the role-switched conversation turn space. The supported roles in Gemini's content schema are `user` and `model` only. Setting `Role: "system"` on a Content used as system_instruction is currently a no-op in the Go SDK (the role field is dropped server-side for the system slot) — but this is undefined behavior that has produced INVALID_ARGUMENT in past SDK revisions, and the published examples consistently leave Role empty for system content. Today it works; nothing guarantees it stays working when the SDK updates the validation.
- **Fix:** Drop the `Role` field on the system Content: `cfg.SystemInstruction = &genai.Content{Parts: []*genai.Part{{Text: prof.SystemInstruction}}}`. Add a regression test that does a real Generate call with a system instruction to catch SDK behavior changes during dependency bumps.

### Single-sample benchmark — Gemini's structured-output nondeterminism makes 9/9 a noisy headline
- **Where:** `benchmark/cases.json`, `benchmark/results/summary.md`
- **Why it matters:** Gemini 2.5 Pro with `responseSchema` is *less* nondeterministic than free-form generation but still not deterministic (temperature defaults to 1.0; bmg never overrides it — verified by grep, `Temperature` is absent from `GenerateContentConfig`). On the cases your own write-up highlights as boundary cases — the 06 readout where Gemini said `~78` (truth 86), the 07 heatmap where it said `~0.1` (truth 0.75), the 08 cluster centroid — a single sample is one draw from a posterior that visibly varies. Publishing "bmg gets these right" or "bmg has a known weakness on 07" from `n=1` is technically inadequate for an OSS release where reviewers will compare against ChartQA-style numbers.
- **Fix:** Run each case 5 times (`temperature=0` if you want determinism for the published table; or 5 samples and report pass@1 mean + std). The actual wall-clock cost is trivial (9 cases × 4 extra samples × ~3s ≈ <2 minutes; pro tokens for 9 images × 4 samples are pennies). For the README, report `mean ± std` or `5/5 passes` instead of a single-shot result. Also: set `Temperature: float32Ptr(0.0)` explicitly somewhere in `Options` (defaulting to 0 for structured-output calls is what every production user of structured Gemini does — leaving it at 1.0 silently hurts you).

### No `temperature` plumbing; defaults to 1.0 from the SDK
- **Where:** `internal/vision/vision.go:155-159` (the `GenerateContentConfig` struct construction)
- **Why it matters:** For schema-constrained extraction tasks (chart values, OCR, graph structure), temperature 1.0 is strictly worse than 0–0.3. Higher temperature increases the chance the model writes a slightly different numeric estimate, picks a different node label paraphrase, or chooses a different `kind` taxonomy entry between calls. The cache layer (sha256 over content) will then thrash because two consecutive describes of the same image return different JSON. This compounds with the no-multi-sample benchmark issue above.
- **Fix:** Add `Temperature *float32` to `vision.Options`, default to `float32Ptr(0.0)` when nil, and pipe to `cfg.Temperature`. For the `general`/hook profile this also tightens cache value-stability — same image → same JSON.

### `intent` is appended to user message, never threaded into `system_instruction` — bypasses the documented separation
- **Where:** `internal/vision/vision.go:147-153`
- **Why it matters:** Your system_instruction docstring on profiles.go:23-29 correctly says system_instruction is for "role consistency without bloating the user-message prompt." But the `intent` text — which is the closest thing to a per-call persona override — is being concatenated onto the user message instead. Google's prompting guidance for Gemini specifically calls out that *task-conditioning hints* (which is what intent is) are most effective when they live alongside the persona priming, not buried in the user message after the structured-output instructions. Empirically, when a system instruction says "respect the schema exactly" and the user message ends with "Prioritize information relevant to this task in your summary," the schema-respect signal often wins and the intent priority gets diluted in the elements ordering.
- **Fix:** Either (a) append intent to `SystemInstruction` instead of `Prompt` when present, or (b) keep current location but prepend intent to the prompt, not append. (a) is the principled fix and gives a measurable bump on cases where intent disambiguates the question — try it on cases 03 (intent disambiguates n1/n2 → content IDs) and 06 (intent forces per-series readout).

## Nice-to-have findings

### `gemini-2.5-flash` for the hook is right; `gemini-2.5-flash-lite` is worth a hidden-flag experiment, not a default
- **Where:** `internal/vision/vision.go:26` (DefaultFlashModel)
- **Why it matters:** flash-lite is ~3x cheaper than flash and the quality delta on simple "general profile, drive-by, no intent" use is small — but lite's OCR is materially worse on small text (sub-100px target font), and the hook is exactly where users won't notice degradation until a UI screenshot gets miscaptured weeks later. Defending the user's hook quality is more important than the ~$0.0003/call savings, so keep flash as the default. Worth exposing `BMG_HOOK_MODEL` env var (bootstrap.md:264 says it's planned — verify it's wired) so power users can opt into lite.
- **Fix:** No code change needed if env override is already wired. Confirm by reading the cmd surface; if not wired, add it before v0.1.0.

### Model name should be configurable, but lock the default for v0.1.0
- **Where:** `internal/vision/vision.go:22, 26`
- **Why it matters:** Locking model names in source for v0.1.0 is fine — Gemini's model release cadence has been steady but unpredictable (2.0-flash → 2.5-flash → 2.5-flash-lite all in <8 months), and a config-overridable model with a sensible default lets you ship without committing to a forever-default. `gemini-2.5-pro` for deliberate and `gemini-2.5-flash` for hook are the right defaults today. `gemini-2.0-flash` as a fallback isn't worth the maintenance — 2.5-flash is strictly better and not more expensive. `gemini-1.5-pro` is similarly obsolete for this use case (worse spatial reasoning, more verbose under structured output).
- **Fix:** Confirm `BMG_DESCRIBE_MODEL`/`BMG_HOOK_MODEL` env overrides exist per bootstrap.md:264. Document in README that model names are intentionally pinned and the upgrade path is "bump constants, re-run benchmark, tag minor version."

### Thinking budget — 128 floor is correct; higher budgets are not free wins
- **Where:** `internal/vision/vision.go:31` (MinProThinkingBudget), and the documented heatmap-at-8192 regression
- **Why it matters:** The 8192-thinking-budget-makes-heatmap-worse observation is **not** a one-off; it's a documented behavior pattern in multimodal LLMs called *overthinking-induced hallucination amplification*. Extended thinking gives the model more tokens to commit to an early wrong interpretation; if the model's first read of a heatmap cell color is off, more thinking tokens elaborate on that wrong color rather than re-examining the pixels. (Same dynamic shows up in Anthropic's extended-thinking ablations on visual reasoning, and in DeepSeek-VL's chain-of-thought ablations.) For *symbolic* tasks (the graph traversal in case 03, the JSON structural extraction in case 09) higher thinking budgets *do* help — they're verbal reasoning, not visual re-perception. The right policy is exactly what bmg does: floor at 128, never let it drift higher without per-profile measurement.
- **Fix:** Add a one-line comment near `MinProThinkingBudget` noting the empirical no-go for higher budgets on perceptual cases ("thinking budget >128 measurably hurt heatmap perception in benchmark; do not raise without per-profile ablation"). Lock the prior so future contributors don't "optimize" this.

### CatmullRom resize is right; consider noting why in the source
- **Where:** `internal/vision/convert.go:73-76`
- **Why it matters:** CatmullRom (a cubic, 4-tap kernel with negative lobes) is the correct choice over BiLinear for chart/diagram content with thin lines. BiLinear blurs single-pixel edges into low-contrast smudges; bicubic-Catmull preserves edge crispness which is exactly what OCR and axis-tick reading need. Lanczos would be a half-step better for very-high-frequency content (small font OCR) but is noticeably slower and the quality gain isn't worth it at the 800px/2576px target sizes bmg uses. The current choice is well-reasoned.
- **Fix:** None required. Optionally extend the source comment: "CatmullRom over BiLinear because thin lines (chart axes, diagram arrows, single-pixel borders) blur into low-contrast smudges with linear interpolation; Lanczos is marginally better for sub-pixel text but ~2x slower at the sizes we target."

### JPEG q70 is defensible for the hook, borderline for the deliberate path
- **Where:** `internal/vision/convert.go:22`
- **Why it matters:** JPEG q70 introduces visible 8x8 DCT artifacts in chroma-subsampled regions, particularly around small text (axis labels at ~9pt rendered fonts) and high-contrast edges (diagram arrowheads on white). At 800px (hook) this is invisible — the resize lossiness dominates. At 2576px (deliberate path), q70 *is* throwing away signal that matters for OCR. The 06 multi-series chart legend is rendered at ~8pt and the legend swatches are 6-8px wide; q70 chroma subsampling smears the color swatch boundaries and is plausibly part of why Gemini sometimes confuses series colors. PNG would not have this issue, but the transparent-PNG → "Could not process image" pipeline reason for the JPEG conversion is real and shouldn't be undone.
- **Fix:** Bump JPEG quality to 85 for the deliberate path while keeping 70 for the hook. Sticky cost change: q85 is ~30-40% larger payload, which costs ~30-40% more *image* tokens (input token count scales with resolution and content complexity in Gemini's vision encoder, not strictly with bytes, but bytes is a reasonable proxy at fixed resolution). Per-call cost increment is still pennies on pro. Two-line change: thread a `JPEGQuality` through `Options` (default to 70, set to 85 in the MCP path), or just hard-code per-path.

### Benchmark needs at least one external dataset point
- **Where:** `benchmark/cases.json` (only 9 synthetic cases)
- **Why it matters:** Reviewers will rightly ask "how does bmg score on ChartQA?" and you have no answer. Synthetic deterministic cases are great for regression testing but useless for external validity — the matplotlib-generated `tab10`-palette charts with clean tick labels are unrepresentative of real-world chart messiness (saturated brand colors, weird fonts, screenshot-of-a-PDF-of-a-chart compression). Suggest reporting numbers on a small slice (~50 examples) of one external benchmark. **ChartQA** is most directly relevant to your `chart` profile, has CC-BY data, and is small enough to run cheaply. **DocVQA** is the matching choice for the `document` profile. **CharXiv** would test the dense scientific figures the 02/07/08 cases gesture at but is harder to score. Skip MMMU and RealWorldQA — too broad, not differential to bmg's value prop.
- **Fix:** Add `benchmark/external/chartqa_subset.py` post-v0.1.0 (not block-release). For v0.1.0, just add a line to `summary.md` acknowledging "external-benchmark numbers tracked in #N (post-0.1.0)."

### Synthetic-images-only is a real weakness — add 2-3 photographed real screenshots
- **Where:** `benchmark/images/`
- **Why it matters:** All 9 cases are matplotlib/PIL synthesis: no JPEG re-encoding, no Retina-scale aliasing, no display gamma weirdness, no camera-phone screenshots, no compressed-by-Slack-twice images. Real bmg traffic is overwhelmingly screenshots-of-screens (sometimes after a screen-recording → frame-export → Slack-paste round trip). Add 3 hand-collected real cases (one phone-camera-of-laptop-screen, one heavily-compressed Slack-relayed screenshot, one Retina-scale UI screenshot from a Mac) and the benchmark goes from "reproducibility theatre" to "real-world plausible."
- **Fix:** Add `benchmark/real/` with 3 hand-collected images and matching questions. Doesn't need to be in v0.1.0 — could be the first post-release issue.

### Profile prompt: `chart` schema has no `chart_type=heatmap` axis handling
- **Where:** `internal/vision/profile_chart.go:13`
- **Why it matters:** The chart profile's schema lists `heatmap` as a valid `chart_type` but the `series`/`axes`/`notable_values` shape is built for line/bar/scatter and doesn't model heatmaps well. A heatmap is a 2D grid of (row, col, value) tuples, not a series with notable values — forcing the heatmap into the series schema is part of why case 07 produces a single bogus `series` with a `notable_values` list of "cells." This is a schema/prompt mismatch and is a contributing factor to the 07 mis-readout (alongside the spatial-grounding limitation called out below).
- **Fix:** Either (a) drop `heatmap` from chart's chart_type enum and route heatmaps to a new `heatmap` profile with `(row_labels, col_labels, cells[][])` schema, or (b) add an optional `cells` field to the chart schema that's populated when chart_type=heatmap. Option (a) is cleaner. Not block-release if you accept that heatmaps are a known weak spot.

### Profile prompt: `code` profile doesn't account for visible line-number gutters
- **Where:** `internal/vision/profile_code.go:16` ("If line numbers are visible in the gutter, do NOT include them in code_text — record those in annotations instead.")
- **Why it matters:** Good that the prompt explicitly addresses this. The annotations array's `kind=gutter_line_number` is a clean shape. One real-world wrinkle: when the screenshot is a *diff view* (commit_view or diff_view context), there are *two* line-number columns (before/after) and the `-`/`+` column is itself a third visual element. The current schema can express this via multiple annotations but the prompt doesn't tell the model to expect dual gutters. Minor.
- **Fix:** Add one sentence to the code prompt: "For diff views, the +/- column and both line-number gutters are separate annotation streams; do not include +/- markers in code_text." Or accept the current behavior — diffs are an edge case.

### Profile prompt: `screenshot` schema is good; `focused_element_label` is the right abstraction
- **Where:** `internal/vision/profile_screenshot.go`
- **Why it matters:** This profile is well-designed. The accessibility-tree framing is correct (matches AT-SPI / NSAccessibility / Chrome's accessibility tree), the `role`/`state`/`is_interactive` triple is the standard one (matches WCAG/ARIA), and `focused_element_label` is the right way to surface "what's the user looking at" without inventing a focus enum that conflicts with `state`. Case 05's output (focused_element_label="error_banner") demonstrates the design working.
- **Fix:** None.

### Profile prompt: `diagram` profile's success on case 03 is not a fluke
- **Where:** `internal/vision/profile_diagram.go`
- **Why it matters:** The "n1/n2 → start/validate/approve" win in case 03 is robust because (a) the system_instruction explicitly says "prefer concise, content-derived IDs over n1/n2" — that's the load-bearing line, and it's a behavioral instruction multimodal LLMs reliably follow under structured output, and (b) the `id` field is `Required` with no enum constraint, so the model is free to pick content-derived strings. The robustness concern is whether this generalizes to diagrams where node text is genuinely ambiguous (two boxes both labeled "process step") — there the model will collide IDs and the diagram will fail to round-trip. Case 09 actually exhibits a related issue: the model emitted edge `from_id=reject, to_id=retry, direction=backward` when the source diagram has `retry → audit, no` (a forward edge from retry to reject was what the truth was, and Gemini got the direction inverted). This is a perception failure, not a prompt failure.
- **Fix:** None for the prompt. Add a fixture diagram with two same-text boxes to surface the ID-collision behavior; document that edge direction is a known weak spot of multimodal LLMs (this is a real, well-known issue in graph-extraction literature — direction inversion on arrow endpoints is ~5-10% error rate even on Gemini 2.5 Pro).

### Spatial grounding below ~100px target — known limitation, not a bmg bug
- **Where:** Case 07 heatmap (cells are ~52x46px on the original render); the cluster-localization case 08
- **Why it matters:** Your maintainer's hunch is right and supported by published numbers. Gemini 2.5 Pro's documented box-2d accuracy degrades sharply for targets below ~100px in any dimension — this is a known limitation of vision-language models that use ViT-style patch embeddings (Gemini uses 14×14 patches, so a 52×46 target spans roughly 3×3 patches and the position embedding resolution at that scale is mush). The mitigation is exactly what the MCP `region=` parameter enables: crop to a ~3-5x neighborhood around the target before sending. For case 07 specifically, `region=[400,600,700,900]` (a centered crop around r5/c7) on the resized image would give the model a ~500×500 view where the target cell is ~150×150px and color-discrimination becomes reliable.
- **Fix:** Document this in `docs/PROFILES.md` (or README) as a known-good pattern: "For dense-figure cases (heatmaps, scatter plots with many overlapping points), call `bmg_describe` first to get the layout, then call again with `region=` to crop to the area of interest." Optionally: a `bmg_describe_region` or `region_hint` autodetection helper, post-v0.1.0.

### Auto-profile routing is premature
- **Where:** General observation; the bootstrap doc mentions `bmg_classify` (line 111) for auto-routing
- **Why it matters:** The right v0.1.0 stance is the current one: caller specifies the profile. Auto-routing requires a meta-classifier (cheap with flash, but still adds latency and a second source of error). Worse, mis-routing is hard to debug from the caller's perspective — they asked for a chart description and got a screenshot one, with no signal why. Caller-specified profile is the principled default; auto-routing belongs in a v0.2 experiment with a confidence threshold and a fallback to `general`.
- **Fix:** None for v0.1.0. Defer `bmg_classify` to v0.2 explicitly in the README's "what's not in v0.1.0" section.

### Two-pass (free-form reasoning → structured extraction) is not worth it here
- **Where:** General observation about structured output
- **Why it matters:** The "structured output hurts reasoning quality" argument is real for free-form *generation* tasks (creative writing, open-ended QA) but is overstated for *extraction* tasks like bmg's profiles. The Gemini 2.5 Pro structured-output decoder runs the same model with a constrained grammar at decode time, not a separate weaker model. The quality cost vs free-form is real but small (~3-5% on chart-extraction tasks per Google's own ablations) and the round-trip stability gain is huge — two-pass would double API cost and latency for marginal quality return. Stay with single-pass structured output.
- **Fix:** None.

### Resolution: Opus's 2576 cap vs Gemini's actual limits
- **Where:** `internal/vision/vision.go:38-44` (DeliberateMaxDim=2576)
- **Why it matters:** Gemini 2.5 Pro's documented image limits are higher than 2576 — the API will accept up to ~3072 on the longest edge before forcing a resize on its side (free tier; paid is the same or higher). The choice of 2576 to match Opus 4.7 makes sense as a *consistency* story (bmg gives users the same upper bound regardless of which Claude model they're on), but for the deliberate path where quality is the priority and the only consumer is Gemini, you're leaving ~20% linear resolution on the table. The 02-08 cases would not benefit (none of them have small-text bottlenecks that 2576 doesn't already solve), but for DocVQA-class workloads (multi-column document scans, dense tables) the difference matters.
- **Fix:** Either (a) raise `DeliberateMaxDim` to 3072 and accept the higher token cost, or (b) document the choice explicitly: "deliberate path caps at Opus's image cap (2576) to keep parity with the alternative path; users wanting full Gemini resolution can set `BMG_DELIBERATE_MAXDIM=3072`." Latter is cleaner.

### Cache key is content-only — good for the hook, slightly wrong for MCP describe with intent
- **Where:** `cmd/bmg/cmd_mcp.go:293` ("content-only — region/intent/profile aren't part of the key")
- **Note:** Content-only cache means two MCP calls on the same image with different intents return the same cached result. ## Summary

The technical choices are mostly defensible and reflect real domain knowledge — CatmullRom resize, JPEG flatten for the PNG-transparency crash, schema-per-profile over mega-schema, thinking_budget=128 floor, model split (pro for deliberate / flash for hook), trust-fenced markdown with OCR isolated as UNTRUSTED. The profile prompts are well-written: each one has an orthogonal system_instruction that primes behavior without duplicating the user prompt, and the schemas have correct `PropertyOrdering` + `Required` lists which Gemini's structured-output decoder rewards. The diagram-profile win on case 03 is real and robust because the system_instruction tells the model to prefer content-derived IDs.

**Top 3 to fix before v0.1.0 tag:** (1) swap image and prompt order in `vision.go:170-173` so text comes first (one-line change, measurably helps task-conditioned cases — Google's own current Gemini docs recommend this order), (2) drop the `Role: "system"` field on `SystemInstruction` Content (one-line change; current value is undefined behavior even if today's SDK tolerates it), and (3) explicitly set `Temperature=0` and run the 9-case benchmark with `n=5` samples per case so the published numbers aren't a single-shot draw from a temperature-1.0 posterior. The heatmap-at-8192 thinking-budget regression you observed is a *real* and well-documented multimodal-LLM behavior (extended thinking amplifies early misperception on perceptual tasks); the current 128-floor policy is correct and should be locked with a source comment so future contributors don't "optimize" it upward.
