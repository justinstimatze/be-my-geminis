## Block-release findings

### `usage()` advertises two flags that don't exist
- **Where:** `cmd/bmg/main.go:92` prints `bmg init [--enable-mcp] [--enable-snap] [--enable-deny-inline]`, but `cmdInit` in `cmd/bmg/cmd_init.go:31-35` defines only `--project` and `--enable-mcp`. `flag.NewFlagSet("init", flag.ExitOnError)` will `os.Exit(2)` on any user who copy-pastes from `bmg --help`.
- **Why it bites the next person:** First-touch contributor types `bmg init --enable-snap` to test the feature they saw in `--help`, gets `flag provided but not defined: -enable-snap`, files an issue. Or worse, ships docs/blog posts referencing the phantom flags. The `--project` flag IS defined and useful but unmentioned in `usage()` — so the usage text is wrong in both directions (over- and under-stating).
- **Fix:** Reconcile `usage()` with the real flagset. Either implement the snap/deny-inline scaffold as `flag.Bool("enable-snap", false, "not yet implemented; reserved")` no-ops that print a "Phase X" message, or just delete them from `usage()` until they ship. Same drift exists in main.go's package doc comment (`cmd/bmg/main.go:9-16` lists `bmg hook deny-inline` which is half-implemented and prints "not yet implemented (Phase 5)" at `cmd/bmg/main.go:80`).

### MCP `inputSchema` advertises `model` but README + MCP server comments + bmg memory describe `max_dim` as a tool arg that doesn't exist
- **Where:** `cmd/bmg/cmd_mcp.go:172-209` (`describeToolDefinition`) declares `path`, `profile`, `intent`, `region`, `model` in the inputSchema. The `args` struct at `cmd_mcp.go:220-227` parses the same five. But the audit prompt and surface description elsewhere references `max_dim` as a parameter — and the README example at `README.md:256-263` documents only `path`, `profile`, `intent`, `region` (no `model`, no `max_dim`).
- **Why it bites the next person:** The downstream agent reads the MCP `tools/list` response to pick which args to send. If a user copies the README example, they cannot override the model — which is the documented escape hatch for "Gemini sunset 2.5-pro". If a developer reads the task spec or the audit prompt expecting `max_dim` as a tool arg, they'll be confused why their override didn't take effect (`cmd_mcp.go:284` always passes `vision.DeliberateMaxDim` and there's no way to override per-call). All three docs (audit spec, README, code) disagree on the MCP tool's argument contract. **This is the API stability surface most likely to break consumers silently.**
- **Fix:** Pick one canonical contract and synchronize. The natural set is `(path required, profile, intent, region, model, thinking_budget, max_dim)`. Add the missing args to `args` struct, `describeToolDefinition`, and `callDescribe`'s wiring; update README example. Then add a single test asserting `describeToolDefinition()` matches a golden JSON file — that test catches every future drift in one place. Also add `bmg_describe` to a `docs/mcp-tool.md` referenced from README so downstream agents have a stable contract to point to.

### Schema "version" string is a static lie — no actual schema versioning exists
- **Where:** `cmd/bmg/cmd_pre_read.go:205` writes `Schema v1` into every report header as a literal: `*Source image: %s · Profile: %s · Generated: %s · Schema v1*`.
- **Why it bites the next person:** When the `chart` schema needs a `heatmap_bins` field (batch 1 flagged this), or when the `diagram` schema needs to add `subgraphs`, the change happens silently — every report still says "Schema v1" because nothing increments. Downstream agents that parse the structured analysis JSON cannot gate on a version field because no version field is emitted. There is no `Schema.Version` constant in the per-profile files; there is no `version` field in any profile schema's JSON shape; there is no `SCHEMAS.md` documenting what v1 means. The string is decorative, not load-bearing — but readers will assume it means something.
- **Fix:** Either (a) remove the string until real versioning lands (least work; honest), or (b) implement it: add `Version string` to `Profile` in `internal/vision/profiles.go`, populate per profile (`"general": "1.0"`, etc.), embed it as a top-level `schema_version` field in every profile's JSON schema so downstream consumers can parse it, and bump per-profile versions independently when adding/renaming fields. A `docs/SCHEMAS.md` enumerating each profile's schema and changelog is the long-term answer. The current commit-and-pray model will silently break the first consumer who writes a typed Go/TS consumer of the structured-analysis JSON.

### No tests for the highest-risk APIs: hook stdout shape, MCP tool definition, vision profile schemas
- **Where:** Test coverage maps:
  - `internal/apikey/apikey_test.go` — 5 tests (env precedence, file fallback, trimming, not-found). Good.
  - `internal/cache/cache_test.go` — 3 tests (put/get roundtrip, Key determinism, Path shape). Adequate.
  - `cmd/bmg/cmd_init_test.go` — 16 tests on settings.json/`.claude.json` merge/unmerge. Excellent.
  - `cmd/bmg/cmd_doctor_test.go` — 8 tests on hook discovery + executable checks. Good.
  - `internal/vision/` — **zero tests**. No test of `Describe`, `ConvertForVision`, profile registration, `walkOCRStrings`, `head()`, or even that all 6 profiles register at init time.
  - `internal/hook/` — **zero tests**. No test that `EmitRedirect` produces JSON of the shape `{"hookSpecificOutput": {"hookEventName": "PreToolUse", "permissionDecision": "allow", "updatedInput": {"file_path": "..."}}}` — the exact contract that broke between Phase 0 attempts on CC 2.1.126.
  - `cmd/bmg/cmd_pre_read.go` — **zero tests**. The decision tree at lines 38-49 (8 branches) has no Go-test coverage. Pressure test 01 catches end-to-end behavior but doesn't pin the exact JSON shape.
  - `cmd/bmg/cmd_mcp.go` — **zero tests**. JSON-RPC parsing, `describeToolDefinition`, `callDescribe`, `cropRegion` are all untested except via pressure test 03.
- **Why it bites the next person:** The test suite is heavily skewed toward the install/uninstall code (which is brittle but reversible) and away from the wire-protocol boundary with Claude Code and downstream MCP clients (which is brittle AND silently breaks). 32 tests sounds great but the smallest end-to-end "tracer" — "does `cmdPreRead` given a JSON stdin payload produce JSON stdout in the shape CC honors?" — does not exist as a unit test. If CC renames `updatedInput.file_path` to `updatedInput.filePath`, only pressure test 05 would catch it, and only if run against a live CC install with the new version.
- **Fix:** Three small wins:
  1. `internal/hook/hook_test.go`: assert `EmitRedirect`/`EmitDeny`/`EmitSessionStart` produce exact JSON byte sequences. Golden files. Five-line tests.
  2. `cmd/bmg/cmd_mcp_test.go`: assert `describeToolDefinition()` matches a golden JSON file. Catches drift between `args` struct and inputSchema (see the `max_dim`/`model` finding above).
  3. `internal/vision/profiles_test.go`: assert exactly 6 profiles register and each has Name/Prompt/Schema non-empty; iterate `Schema.Properties` and assert the `Required` slice covers every property (the "every field required" pattern is the design — make it enforced).

### No `CHANGELOG.md`, no `CONTRIBUTING.md`, no schema-evolution / API-stability policy documented
- **Where:** Repo root: only `README.md`, `LICENSE`, `bootstrap.md`. `main.go:121` references `CHANGELOG.md` ("See README.md for usage and CHANGELOG.md for release notes.") — the file does not exist. Batch 1's security panel already flagged the missing `SECURITY.md`; this is the maintainability companion.
- **Why it bites the next person:** A contributor adding a 7th profile (or evolving `chart` to add `heatmap_bins`) has no documented process. Is bumping a profile schema a minor-version bump for bmg? Does it require a `Schema v2` literal? Does it need a migration path for cached reports written under v1? The answer is "ask Justin" — fine while bmg has one contributor, untenable post-public. The same applies to the `Report` struct in `internal/vision/vision.go:64-77` — if `Elements []Element` is renamed, every external Go consumer of `bmg describe`'s JSON output breaks (the `bmg describe` CLI prints `elements` as a top-level JSON key at `cmd_describe.go:98`, so this IS a public contract whether bmg likes it or not).
- **Fix:** Before v0.1.0:
  1. Add `CHANGELOG.md` (Keep-a-Changelog format; v0.1.0 entry).
  2. Add `CONTRIBUTING.md` covering: how to add a profile, schema-evolution policy (additive within minor, breaking-rename within major), how to regenerate `bmg doctor` golden output, the `BMG_HOOK_PROXY=1` recursion-guard contract for shell-out hook authors.
  3. Add a short "API stability" section to README enumerating what IS public: the MCP tool's args + the hook output JSON shape + the per-profile structured-analysis JSON shapes. What is NOT public: anything in `internal/`, the `Report` struct, the env-var names other than the four documented (`BMG_API_KEY`, `BMG_CACHE_DIR`, `BMG_DISABLE`, `BMG_HOOK_MODEL`). The internal/ Go boundary already prevents cross-module imports — that's good — but `BMG_HOOK_PROXY=1` is undocumented in README and is load-bearing for anyone writing a shell-wrapper hook in front of bmg.

## Nice-to-have findings

### Two model-name constants hardcoded; no runtime detection or graceful sunset path
- **Where:** `internal/vision/vision.go:22, 26` — `DefaultProModel = "gemini-2.5-pro"` and `DefaultFlashModel = "gemini-2.5-flash"`. Referenced from `cmd_pre_read.go:129`, `cmd_describe.go:35`, `cmd_mcp.go:257, 268`, `cmd_doctor.go:79`.
- **Why it bites the next person:** Google has sunset Gemini models on roughly 6-month cycles (1.0-pro → 1.5-pro → 2.0-pro → 2.5-pro, with formal deprecation notices ~3 months before retirement). When 2.5-pro is sunset, the user experience is a cryptic `404 model not found` from `client.Models.GenerateContent` propagated through `denyWithReason` as `Gemini describe failed: …`. `bmg doctor` will fail its ping check — that's the saving grace — but the doctor message will just blame Gemini, not name the sunset. There is no `bmg upgrade-models` migration story, no version-pin in `~/.config/bmg/config.toml` (no config file exists), no per-request fallback chain (`try 3.0-pro → fallback to 2.5-pro`).
- **Fix:** Two layers. (1) Add a top-of-file comment in `vision.go` linking to Google's deprecation schedule + the date bmg's defaults were last validated. (2) When `Ping` fails with a 404-style error, have `bmg doctor` detect the substring "model" + "not found" or "deprecated" and print a specific upgrade hint pointing at the README's "Configuration" section listing `BMG_HOOK_MODEL` / `BMG_DESCRIBE_MODEL` overrides. The current `Ping` just returns the raw genai error.

### `genai` SDK v1.55.0 pinned with no documented update cadence
- **Where:** `go.mod:7` — `google.golang.org/genai v1.55.0`. The SDK has been on the 1.x line for ~1 year and goes through minor versions every few weeks (this is v1.55.0 — that's ~55 minor releases). Historical examples: `Schema` vs `ResponseSchema` vs `StructuredOutput` field renames, `ThinkingConfig.ThinkingBudget` changing from `int32` to `*int32`. Both impacted bmg's surface during Phase 0 per the memory notes.
- **Why it bites the next person:** Quarterly `go get -u google.golang.org/genai` works until it doesn't. The single test most likely to catch breakage (a real `genai.NewClient(...)` call) only runs in pressure test 05 — and pressure tests are documented as "ungated on the next dev's machine" per the README's Development section.
- **Fix:** (1) Pin via `// renovate: pin` comment or document update cadence in CONTRIBUTING.md. (2) Add a `go test -tags=network ./internal/vision/...` target that runs a real (cheap, 1-token) `Ping` against Gemini. CI doesn't need it but a maintainer reviewing a SDK bump should run it. (3) Wrap the genai-specific types (`*genai.Schema`, `*genai.ThinkingConfig`) behind a thin internal type so a future SDK rename is a single-file change in `internal/vision/`.

### CC hook fields are documented in code comments but not as a contract
- **Where:** `internal/hook/hook.go:35-44` (Input struct), `:60-69` (PreToolUseOutput), `:99-109` (SessionStartOutput). All documented with "Verified empirically against Claude Code 2.1.126" comments — solid for "I remembered to leave a breadcrumb", weak for "the next person can validate against CC 2.2.0".
- **Why it bites the next person:** When CC 2.2.0 changes the SessionStart hook to pass `additionalContext` via a different JSON path (or removes the `hookEventName` requirement entirely), bmg silently produces output CC ignores. The SessionStart hook is the most fragile: it has no return-value validation — if CC silently drops the `additionalContext` because the shape changed, Claude continues without the routing-instructions prime and the OCR trust-fence model collapses without anyone noticing until a prompt-injection lands.
- **Fix:** Add a `pressure_test/05` sub-case that round-trips through CC. Specifically: write a script that spawns `claude --help` or some no-op CC invocation, parses the hook events CC actually emits today, and asserts our `Input` struct unmarshals each key without losing a field. Bonus: a "supported CC versions" badge in README that humans update when they verify against a new version. The pressure tests already exist for the bmg side of the contract; CC's side is untested.

### Vision package's `Report` struct is doubly-purposed (typed Elements only for general; Raw for everything else)
- **Where:** `internal/vision/vision.go:64-77` — `Summary` is per-profile; `Elements` is general-only; `Raw` is all profiles' verbatim JSON. The comment at line 58-63 explains this but the type signature alone is ambiguous. `cmd_describe.go:98-99` emits both `elements` and `structured` keys, where `structured` is the parsed `Raw` — that's a hint to the consumer that `elements` is legacy.
- **Why it bites the next person:** A contributor writing a 7th profile will look at `Elements` and either (a) assume they should populate it (wrong; only `general` does) or (b) ignore it and write their own typed accessor in their profile file (creating drift). The struct doesn't enforce the design: `Elements` should logically be inside the general profile's typed accessor, not on the shared `Report`.
- **Fix:** Two options. (a) Move `Elements` off `Report` entirely; provide `(Report).AsGeneral() (*GeneralReport, error)` that unmarshals `Raw` into the general-specific typed struct. Same pattern for `(Report).AsChart()`, `(Report).AsDiagram()`, etc. (b) Cheaper: rename `Elements` to `GeneralElements` and add a one-line comment // GeneralElements is populated ONLY when Profile == "general". Either way, document that the shape is a transitional artifact of Phase 1 and the typed-per-profile accessor is the long-term API.

The `cmd/bmg/` directory pattern is already "one file per subcommand" — make it "one file per install target" within `init`. This is the only file in `cmd/bmg/` that violates the existing organizational pattern.

## Summary

Top three brittle externals to harden before v0.1.0: (1) **MCP tool inputSchema is the most visible contract** and it disagrees with the README example (`max_dim` referenced in spec but absent from code; `model` exposed in code but absent from README) — add a golden-file test in `cmd/bmg/cmd_mcp_test.go` and reconcile. (2) **Hook stdout JSON shape** has no Go-level test pinning the exact contract bmg has with CC; one `internal/hook/hook_test.go` with five-line golden assertions would catch the silent-breakage case where CC renames `updatedInput.file_path`. (3) **Gemini model names + SDK pin** have no documented update cadence and `bmg doctor` doesn't distinguish "model deprecated" from generic Gemini errors — a comment in `vision.go` linking to Google's deprecation schedule plus a sunset-substring check in doctor closes the gap cheaply.

Top three maintainability issues that bite the future contributor first: (1) **`usage()` advertises `--enable-snap`/`--enable-deny-inline` flags that don't exist** and omits the `--project` flag that does — a contributor's first `bmg init --enable-snap` exits 2 with a confusing flag-not-defined error; this is the first thing they'll touch and the first thing that breaks their trust in the docs. (2) **No `CHANGELOG.md`, `CONTRIBUTING.md`, or schema-evolution policy** combined with the lying `Schema v1` literal means there's no documented answer for "I want to add a heatmap field to chart" — the per-profile JSON schemas are public contracts that downstream agents parse, and there's no story for how they evolve. (3) **Zero tests on `internal/vision/`, `internal/hook/`, `cmd/bmg/cmd_pre_read.go`, `cmd/bmg/cmd_mcp.go`** — 32 tests sounds great but the test mass is on install/uninstall (reversible bugs) while the wire-protocol APIs (silent-breakage bugs) are unprotected; three small test files covering profile registration, hook stdout JSON, and MCP tool definition would shift the safety profile dramatically.
