## Block-release findings

### 1. The README cherry-picks the four cases where bmg looks best and silently drops the four where it loses or ties

- **The objection (as it would land in a top comment):**
  > "9 cases in `benchmark/cases.json`, but the README shows 4 (cases 01, 03, 04, 05). The four it left out — 02, 06, 07, 08, 09 — happen to be the ones where raw Opus answers the question cleanly or where bmg's answer is just wrong. You can't say 'structurally better' and then publish the four cases where bmg has the most structure to wave around. Show me the full table or this is marketing."
- **Why it matters:** Anyone with five minutes will diff README cases against `cases.json` and notice the gap. The first link to `benchmark/results/summary.md` (line 82 of README) gives them the ammunition. The story goes from "principled engineering" to "salesman" in one click. The repo's own commit log shows case 07 was deliberately reworded to "defeat ambiguity" — that's a signal it caused trouble, and it's still not in the README.
- **Fix:** Either (a) show all 9 in the README with the losses included and a one-sentence honest readout per case (`Opus correct, bmg correct, bmg adds structure` vs `Both correct, equivalent` vs `bmg wrong here, here's why we still publish it`), or (b) drop the README cases entirely and write `"see the full 9-case run in benchmark/results/summary.md — bmg wins on structure in 4, ties on prose in 3, is wrong on 2 (06, 07)"`. The honesty is the moat.

### 2. Case 06 is in the benchmark and bmg gives the wrong answer — README hides it

- **The objection (as it would land in a top comment):**
  > "Pulled the benchmark. Case 06 asks for memory_kb at t=42. The generator literally bakes in `ys = [50 + t * 0.85]`, so the answer is 85.7. Raw Opus computed 85.6. bmg returned `~78` and confidently wrapped it in a JSON schema. So your pitch is 'structured output you don't have to parse' but the structured field has a wrong number? At least the prose answer made the error visible. Schema-wrapped wrong is worse than prose wrong, because it disables Claude's skepticism."
- **Why it matters:** This is the single most damaging point on the front page. It directly inverts the pitch. "Typed output" is supposed to be the advantage; in case 06 the typed output is wrong, the typed output is more confident than the prose, and Opus (the thing being replaced) beat it on the same image. If this lands in a top comment with the numbers, the conversation is over.
- **Fix:** Three options in order of integrity: (a) acknowledge the regression in the README — call it the limit of vision LLMs on dense-line charts, regardless of vendor; (b) add a "where bmg loses" section that names cases 06 and 07 and the failure mode; (c) drop "structurally better" framing and replace with "structurally typed — accuracy is still bounded by the underlying vision model, and Pro is currently better on extraction tasks but not on per-pixel readouts off a 6-line chart." Then publish it. Do not ship without one of these.

### 3. Case 07 was rewritten to "defeat ambiguity" — that's exactly the disclosure HN will want

- **The objection (as it would land in a top comment):**
  > "Looked at `cases.json` for case 07. The `what_it_tests` field literally says 'Question is worded to defeat the row 5 (0-indexed) vs 5th row (1-indexed) ambiguity that earlier-version 0-indexed-position phrasing surfaced.' Translation: 'we changed the question after seeing the first run.' That's the textbook definition of p-hacking a benchmark. How many other questions got tuned? Was Opus or bmg the one failing on the original phrasing?"
- **Why it matters:** The benchmark file is in the repo, the rewording is documented in plain English in the data. A skeptical reader will quote the line verbatim. Even if the rewording was honest disambiguation, the optics destroy the benchmark's credibility unless preemptively explained.
- **Fix:** Add a one-paragraph "Benchmark provenance" note at the top of `benchmark/README.md` and `benchmark/results/summary.md`: which questions were reworded, what the original phrasing was, which model failed on the original, and why the new phrasing isn't a leak. The bake-in `grid[5,7]=0.75` in `generate.py` (line 290) where bmg answered "~0.1" also needs to be acknowledged — bmg is wrong by a factor of 7.5x on a question with a known ground truth, and the README doesn't mention this case exists.

### 4. Zero privacy disclosure in the README — "every image you Read goes to Google" is buried in design docs, never in user-facing copy

- **The objection (as it would land in a top comment):**
  > "Install this and every screenshot, every NDA'd design mockup, every customer support image, every PDF page with PII you ask Claude to look at gets uploaded to Google's API by a hook that fires on every Read. The README's three-sentence install section doesn't say this. No mention of Google's data retention. No mention of whether you're paying for the no-training tier. No GDPR note. No SOC2 note. For an enterprise dev this is a non-starter on the spot."
- **Why it matters:** This is the second-most-likely top comment after the benchmark issue. The Be-My-Eyes framing makes it worse — that app has explicit privacy controls because of the obvious sensitivity; bmg has none. Searched the README and bootstrap.md for `privacy`, `gdpr`, `soc2`, `retention` — zero hits in README, only `BMG_BUDGET_USD` warnings in bootstrap.
- **Fix:** Add a "Data flow / privacy" section to the README, above install. Three bullets: (1) every image read by Claude Code is sent to Google's Generative AI API by default; (2) link to Google's current AI Studio / Vertex retention policy and explicitly state which tier the user is hitting based on which env var they set; (3) name the kill switches — global (`bmg uninstall`), per-session (`BMG_DISABLE=1` at CC launch), per-Read (acknowledge this is impossible mid-session without setting the var before launching CC — see finding #5). Without this section, this project will not be deployable inside any company with a GRC team.

### 5. `BMG_DISABLE=1` "for one Read" is misleading — mid-session there's no per-Read opt-out

- **The objection (as it would land in a top comment):**
  > "README line 310: '`BMG_DISABLE=1` in environment bypasses the hook for one Read.' But the hook is a PreToolUse process that inherits the CC parent's env. You can't change the env of a running CC session for the next Read only. So in practice this is per-session, set at launch — not per-Read. The wording is wrong, and it matters because the only mid-session escape is `bmg uninstall` + restart, which kills the rest of the session."
- **Why it matters:** Operationally important and concretely wrong. The first user to read this carefully will post a correction. The lack of a mid-session escape combined with "every image goes to Google" (finding #4) is a compounding objection.
- **Fix:** Reword to: "`BMG_DISABLE=1` set in the environment before `claude code` launches bypasses the hook for the entire session. There is no per-Read mid-session toggle; you must restart CC without the hook to opt out. To make a single Read bypass bmg, exit, set `BMG_DISABLE=1`, restart, do the Read, exit, unset, restart." If that's too painful, add a hot-disable flag-file mechanism (`/tmp/bmg-disable-once`) that the hook checks and deletes, and document it.

### 6. Cost claims are off by at least 1.5-2x once realistic usage compounds

- **The objection (as it would land in a top comment):**
  > "README says ~$0.0005 per drive-by hook and $0.02-0.10 per Pro call. Let's add it up. A normal day of CC: I `Read` 30 screenshots (debug, design refs, paste from a doc). That's 30 × $0.0005 = $0.015/day on flash. Fine. But the hook fires whether I wanted Gemini's opinion or not — so I'm paying ~$5/year just to not see images natively. And if Claude calls `bmg_describe` deliberately at $0.05/avg, that's another $0.05 every time it 'helps.' Over a year of heavy CC use this is $20-50, not 'cents.' It's not breaking the bank, but 'cache deduplicates so re-reads are free' implies the marginal cost is zero — it isn't. A first-read fires every time."
- **Why it matters:** Not catastrophic, but the math is checkable and the framing is a touch underclaimed. HN cost-sensitivity is high. The bigger framing problem is the "you didn't ask for this" angle — bmg charges the user for a vendor swap they may not have wanted on a per-image basis.
- **Fix:** Replace the FAQ cost section with a worked example based on a measured week: "In our internal usage of N reads / week, total Gemini spend was $X. The hook fires on every distinct image SHA; cache deduplicates exact rereads. The deliberate MCP path is opt-in by Claude and runs Pro." Add the `bmg doctor` cost report (bootstrap.md says it's planned) so users can see their own number. Until then, do not say "cents."

### 7. "Why not just use Claude Opus 4.7's native vision?" answer is defensive and undercuts the pitch

- **The objection (as it would land in a top comment):**
  > "Read the FAQ. 'On simple synthetic cases Opus 4.7 is genuinely good' — so you've already conceded the strongest case. 'bmg's edge is the typed output shape (no prose parsing), prompt-injection resistance, and Gemini-2.5-Pro's edge on dense / scientific imagery (see CharXiv-style cases).' Two of those three are bmg-the-wrapper claims, not bmg-vs-Claude claims. Typed output: I can ask Claude for JSON, it'll give me JSON. Prompt injection: a real concern, but bmg's defense is a fence-string Claude is asked to honor, which is a soft barrier any sufficiently adversarial image can attack equally well. The only Claude-vs-Gemini claim left is CharXiv-style figures — and your benchmark doesn't have a CharXiv-style figure that bmg unambiguously wins, except by adding bbox coordinates Opus would happily emit if asked."
- **Why it matters:** This is the framing critique that gets the most upvotes because it's a recurring shape of comment ("you didn't need a new project for this"). The README doesn't punch back hard enough. The two-surface (hook + MCP) story is the actually-interesting argument and it's buried.
- **Fix:** Lead with the two-surface architecture, not the cherry-picked side-by-side. The pitch is "Claude has to remember to ask for vision in a specific format — bmg makes vision-with-task-aware-structure the default by hooking the Read path." Then the cases are illustrative, not load-bearing. The current README inverts this.

### 8. The hook fires on every image Read — there is no warning that this includes images the user pasted but didn't want analyzed

- **The objection (as it would land in a top comment):**
  > "If I paste a screenshot of my password manager to ask Claude an unrelated question about the visible app behind it, that image now goes to Google. If I drag in a PDF of a contract to ask 'is page 4 the one I should look at,' the whole page goes to Google including pages 1-3 in the same Read. There's no scope control. The opt-in is at the CC-session level, not at the image level."
- **Why it matters:** This is the privacy concern's evil twin — even an informed user can't selectively opt out per image without restarting their session. For users in regulated industries (medical, legal, financial), bmg becomes uninstallable on these grounds alone.
- **Fix:** Implement a flag-file mechanism (`/tmp/bmg-skip-next`) the hook checks before any API call, plus a `bmg pause [duration]` CLI that writes a TTL file. Document both prominently. Long-term: a CC slash command `/bmg-pause` would be ideal but requires CC-side support.

## Nice-to-have findings

### 9. Case 03's "structural win" is solvable with one line of prompting to Opus

- **The objection:** "Case A shows `start/validate/approve/end` as bmg's node IDs vs Opus's `n1/n2/...` — but Opus answered the question correctly with `start → validate → approve → end`, which is the same content-derived labels bmg used. The 'structural win' here is just: bmg asked Gemini for JSON, Opus was asked a natural-language question. If you ask Opus 'return this as JSON nodes/edges with content-derived IDs,' you get the same thing. The diff is a prompt difference, not a model difference."
- **Why it matters:** Easy to point out, undercuts the headline example.
- **Fix:** Mention this honestly in the README — "if you ask Opus directly for JSON, it can produce the same shape. bmg's pitch is that you don't have to ask, and that the typing is enforced by a responseSchema so the shape doesn't drift."

### 10. No `Tested with Claude Code <version>` matrix; README claims "works on 2.1.126" only

- **The objection:** "PreToolUse hooks have had behavior changes between CC minor versions (the `mcp_tool` hook is documented but ignored — your own memory file `mcp_tool_hook_dead_end.md` says so). What versions has bmg actually been verified on? What breaks on the prior or next minor?"
- **Why it matters:** Hooks are fragile; CC versions move fast.
- **Fix:** Add a table to README: `Last tested green: 2.1.126 / Last known broken: 2.1.108 (issue: ...) / Untested above: 2.1.127+`. Run a CI matrix if possible.

### 11. The "trust fence" is a string convention Claude is asked to honor, not a structural guarantee

- **The objection:** "`<bmg-vision-report origin="gemini-2.5-pro" trust="evaluator-output">` is just a fenced markdown block. Claude's only reason to treat it as trusted is the SessionStart injection telling it to. A sufficiently confused Claude (or one whose context window has rotated past the SessionStart prime) will treat the contents however the local context suggests. This isn't anti-prompt-injection; it's hopeful labeling. Call it what it is."
- **Why it matters:** Security-minded reader catches the gap.
- **Fix:** README is mostly honest about this ("isn't a guarantee — structural barrier missing from the raw-image path") but should add: "rotation through compaction can drop the SessionStart prime; re-run `bmg doctor` and consider `/clear` for long sessions."

### 12. The "Be My Geminis" name riffs on a disability-services org without giving anything back

- **The objection:** "Naming a developer tool after a service for blind people is fine, but the README link to bemyeyes.com is a one-liner under 'About the name' with no donation prompt, no credit-where-due gesture, no 'we'd love it if you donate to them.' If you're going to ride that brand recognition, contribute to the actual cause."

### 13. No prior-art reference — at minimum justi's gist on CC vision hooks should be cited

- **The objection:** "Is this the first project to do PreToolUse:Read → vision API? `bootstrap.md` (line 339) references `docs/PRIOR_ART.md` and a `justi gist` but that file doesn't exist in the repo. Either cite it or claim novelty and own that claim."

### 14. CI matrix is single-version, no integration tests, no real Gemini call exercised

- **The objection:** "`.github/workflows/ci.yml` runs `go test -race`, `go vet`, `go build` on `go-version: 1.26`. No integration tests, no MCP roundtrip test, no Gemini smoke test. The 32 tests in `go test ./...` are unit tests on schema validators. The whole thing could be broken end-to-end and CI would stay green."
- **Why it matters:** Implies the "32 tests" claim is overstated as evidence of quality.
- **Fix:** Add an integration job behind a secret (`if: secrets.GEMINI_API_KEY`) that runs one real Gemini call per profile. Or be explicit in README: "32 unit tests; integration testing is manual via `pressure_test/`."

## Summary

The pitch survives the first hour on the front page only if findings #1 and #2 are fixed before posting — the cherry-picked benchmark and the wrong answer in case 06 are the two objections that will define the thread. If those land unanswered, the top comment will be "you hid the cases where Opus won, and the one new case you'd want to brag about, bmg got wrong with more confidence than Opus did" — and that comment will be screenshot-quotable from the repo's own data. Findings #3 (case 07 rewording), #4 (zero privacy disclosure), and #5 (BMG_DISABLE wording wrong) are the next tier and will compound into "this isn't ready for enterprise users" by hour two. The architecturally-honest pitch — two surfaces, schema-typed output, ambient meta-loop — is real and defensible; the README leads with the wrong artifact (cherry-picked side-by-sides) and buries the actually-novel claim (the hook-as-default-vision-substrate). Fix the benchmark honesty problem, add a privacy section above install, soften the cost claim, and the project lands as "interesting infrastructure experiment, here's where it's weak." Ship as written and the thread eats it.
